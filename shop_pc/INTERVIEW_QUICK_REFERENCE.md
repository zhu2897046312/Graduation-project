# 面试快速参考 - 技术难点一句话总结

## 🎯 四大核心技术难点（30秒版本）

### 1. Naive UI SSR 样式延迟问题
**问题**：Nuxt 3.12 移除 `ssrContext.styles`，导致 Naive UI 样式无法在 SSR 时注入  
**解决**：使用 `@css-render/vue3-ssr` 手动收集样式，通过 `nuxtApp.ssrContext.head.push` 注入  
**价值**：消除 FOUC，提升 SEO 和用户体验

### 2. UUID 服务端客户端不一致
**问题**：SSR 环境下服务端和客户端生成不同 UUID，设备识别失败  
**解决**：使用 Cookie 持久化 + 服务端模块级变量缓存  
**价值**：正确识别游客身份，支持购物车限制等业务

### 3. 邮件发送功能实现
**问题**：从 Resend 第三方服务迁移到自建邮件服务  
**解决**：后端使用 `gopkg.in/gomail.v2`，前端实现密码重置流程  
**价值**：降低成本，提升系统自主可控性

### 4. PayPal 支付集成
**问题**：需要对接 PayPal API，实现完整支付流程  
**解决**：设计订单创建→支付授权→订单捕获流程，与后端协作实现  
**价值**：支持国际支付，扩大用户群体

## 💬 面试常见问题回答模板

### Q1: 介绍一下你的项目
**回答**：这是一个基于 Nuxt 4 的电商平台，我主要负责前端开发。项目最大的挑战是解决了 Nuxt 3.12 升级后 Naive UI 的 SSR 样式问题，以及服务端客户端数据一致性问题。还完成了 PayPal 支付集成和邮件发送功能。

### Q2: 你遇到的最大技术难点是什么？
**回答**：最大难点是 Naive UI 的 SSR 样式问题。Nuxt 3.12 移除了 `ssrContext.styles`，导致样式无法在服务端注入，出现 FOUC。我通过研究框架源码和 GitHub Issue，使用 `@css-render/vue3-ssr` 手动收集和注入样式，完美解决了问题。

### Q3: 你如何保证 SSR 的数据一致性？
**回答**：主要解决了 UUID 生成问题。使用 Cookie 持久化设备指纹，服务端使用模块级变量缓存，确保同一请求中 UUID 一致。客户端从 Cookie 读取，保证服务端和客户端使用相同的设备标识。

### Q4: 你如何集成第三方服务？
**回答**：以 PayPal 为例，我设计了完整的支付流程：订单创建→获取支付链接→跳转支付→回调捕获。与后端协作，使用 PayPal REST API v2，实现了订单创建和支付解耦，提升了系统灵活性。

### Q5: 你的 SEO 优化经验？
**回答**：
1. 解决了 SSR 样式问题，确保搜索引擎抓取时页面完整
2. 实现了动态 TDK 配置
3. 实现了面包屑导航
4. 使用 ISR 平衡 SEO 和性能
